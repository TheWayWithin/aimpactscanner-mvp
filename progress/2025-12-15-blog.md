# Breaking Free from the 60-Second Prison: How I Future-Proofed AImpactScanner

*December 15, 2025*

Today I shipped one of the biggest architectural changes to AImpactScanner since launch. The kind of change that doesn't add flashy features but fundamentally transforms what the product can become.

I migrated from Supabase Edge Functions to a Railway Node.js backend. And in doing so, I removed the invisible ceiling that was limiting everything.

## The Problem Nobody Talks About

Here's what most developers don't realize about serverless edge functions: they have hard timeout limits.

Supabase Edge Functions? **60 seconds max.**

For analyzing a simple static website, that's fine. My analysis engine typically completes in 15-30 seconds. No problem.

But modern websites aren't simple. They're React SPAs. Vue applications. Next.js with client-side rendering. Angular dashboards.

Try to fetch these sites with a simple HTTP request and you get... nothing. An empty `<div id="root"></div>` with a promise that JavaScript will eventually fill it in.

To analyze these sites properly, you need a headless browser. Puppeteer. And Puppeteer needs time to:

1. Launch a browser instance
2. Navigate to the page
3. Wait for JavaScript to execute
4. Wait for network requests to settle
5. Extract the rendered HTML
6. Run the actual analysis

That's easily 90-120 seconds for complex SPAs. Well beyond the 60-second limit.

## The Invisible Ceiling

This timeout wasn't just a technical limitation. It was a business constraint hiding in plain sight.

My target market is SaaS founders. Indie hackers. Startup builders.

**What are they building?** React apps. Vue dashboards. Modern JavaScript frontends.

**What couldn't I analyze?** React apps. Vue dashboards. Modern JavaScript frontends.

I was building an AI visibility scanner that couldn't scan the very websites my target customers were building.

## The Migration

Moving from Edge Functions to Railway wasn't a simple port. It was a complete architectural rethink.

### Before: The Edge Function Architecture

```
Frontend (Netlify) → Edge Function (Deno, 60s limit) → Supabase DB
```

Simple. Fast to deploy. But fundamentally limited.

### After: The Railway Architecture

```
Frontend (Netlify) → Railway API (Node.js, no limits) → Supabase DB
                              ↓
                     Background Workers
                              ↓
                     Puppeteer (for CSR sites)
```

The new architecture includes:

- **Express server** with proper middleware (CORS, auth, rate limiting, logging)
- **Async job queue** for long-running analyses
- **Background workers** ready to process Puppeteer renders
- **Feature flags** for gradual rollout

### The Job Queue Design

For sites that need headless browser rendering, I built a proper job queue:

```sql
CREATE TABLE analysis_jobs (
  id UUID PRIMARY KEY,
  analysis_id UUID REFERENCES analyses(id),
  status job_status DEFAULT 'pending',
  priority INTEGER DEFAULT 0,
  attempts INTEGER DEFAULT 0,
  max_attempts INTEGER DEFAULT 3,
  ...
);
```

The clever bit? A PostgreSQL function that claims jobs atomically:

```sql
SELECT * FROM analysis_jobs
WHERE status = 'pending'
ORDER BY priority DESC, created_at ASC
LIMIT 1
FOR UPDATE SKIP LOCKED;
```

`FOR UPDATE SKIP LOCKED` prevents race conditions when multiple workers try to claim the same job. It's the kind of detail that separates "works in development" from "works at scale."

## What This Unlocks

With the Railway backend live, here's what becomes possible:

### 1. Client-Side Rendered Website Analysis

React, Vue, Angular, Svelte - any JavaScript-heavy site can now be properly analyzed. The headless browser waits for the content to render before extracting it.

### 2. Deep Crawl Capabilities

No timeout means I can crawl multiple pages per analysis. Homepage, about page, pricing page, blog posts - a comprehensive site audit instead of single-page snapshots.

### 3. Competitor Analysis

Queue up multiple sites for analysis and compare them. "How does my AI visibility compare to my top 5 competitors?" That's a feature that requires long-running batch processing.

### 4. Real-Time Progress Updates

The job queue integrates with Supabase real-time subscriptions. Users can watch their analysis progress in real-time, even for multi-minute Puppeteer renders.

## The Transition Strategy

I didn't flip a switch and hope for the best. The migration followed a careful pattern:

1. **Build in parallel** - Railway backend running alongside Edge Functions
2. **Feature flag control** - `VITE_USE_RAILWAY_BACKEND` environment variable
3. **Staging validation** - Full testing on develop branch first
4. **Gradual rollout** - Production enabled after staging verification
5. **Fallback ready** - Edge Functions still deployable if needed

Today I enabled the feature flag in production. The first real analysis (stackoverflow.com) completed successfully: 73/100 score, 19 factors analyzed, all in under 30 seconds.

## The Lesson

But they *will* notice when I can analyze their React SPA that previously returned empty results. They *will* notice when I can show them how their site compares to competitors. They *will* notice features that were simply impossible before.

The best infrastructure work is invisible. It just makes everything else possible.
